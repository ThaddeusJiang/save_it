# 001 Typesense Saved Content

Typesense saved content indexing specification.

## Purpose

Define the durable Typesense document shape and metadata extraction rules for
media saved from Telegram messages. Saved content must keep user-provided
message text separate from URL-derived page metadata so search, detail views,
and future provider-specific extraction can rely on stable field semantics.

## Scope

- In scope:
  - The `photos` Typesense collection document fields used for saved photos,
    videos, URL downloads, URL thumbnail fallbacks, and directly uploaded
    Telegram media.
  - Default field sources for `caption`, `title`, `description`, `keywords`,
    and thumbnail image data.
  - Provider-specific extraction hooks for supported URL platforms.
  - Search indexing expectations for user text and URL metadata.
- Out of scope:
  - Local cache filename policy.
  - Google Drive storage layout.
  - Telegram message delivery formatting, except where Telegram responses are
    the source of persisted index metadata.

## Persistence

### Entities

- `Typesense photos document`
  - `id`: Typesense-generated document id.
  - `image`: required image bytes used by Typesense image embedding. This is the
    saved photo itself, a video thumbnail, or a fallback preview image.
  - `image_embedding`: generated by Typesense from `image`; excluded from normal
    read responses.
  - `caption`: optional string. User-provided text that was sent together with
    the Telegram URL, photo, or video. URL text itself is not part of `caption`.
    URL-derived `og:title`, `og:description`, or `meta keywords` must not be
    stored in this field.
  - `title`: optional string. Defaults to the source URL page `og:title`.
  - `description`: optional string. Defaults to the source URL page
    `og:description`.
  - `keywords`: optional string array. Defaults to the source URL page
    `meta[name="keywords"]` values after trimming empty entries.
  - `url`: optional string. Original source URL when the save came from a URL.
  - `download_url`: optional string. Resolved media URL when it is safe and
    useful to expose.
  - `thumbnail_url`: optional string. Public preview image URL, such as an
    Open Graph image URL. Tokenized Telegram file URLs must not be stored here.
  - `file_id`: optional string. Telegram file id for the saved message media
    that can be sent back to Telegram.
  - `media_type`: optional string. Expected values are `photo` and `video`.
  - `belongs_to_id`: required string. Telegram chat id that owns the document.
  - `source_message_url`: optional string. Jump URL to the original Telegram
    message when Telegram exposes a valid public or private supergroup URL.
  - `inserted_at`: required int64 Unix timestamp used as the default sort field.

### Field Source Rules

- `caption`:
  - Read from the Telegram message that triggered the save.
  - For URL text messages, read `message.text`, remove every URL, trim the
    remainder, and store the result.
  - For Telegram photos and videos, read `message.caption`. If the caption also
    contains the source URL, remove the URL and store the remaining user text.
  - Store an empty string or omit the field when the user did not provide text.
  - Bot command captions such as `/similar` and `/search` are command input, not
    saved user captions.
- `title`:
  - Default to URL metadata `og:title`.
  - Leave unset when no URL metadata exists or the source page has no title.
- `description`:
  - Default to URL metadata `og:description`.
  - Leave unset when no URL metadata exists or the source page has no
    description.
- `keywords`:
  - Default to URL metadata `meta[name="keywords"]`.
  - Split comma-separated keyword content, trim whitespace, remove blank values,
    and preserve provider order.
  - Leave unset when no URL metadata exists or the source page has no keywords.
- `image`:
  - For Telegram photo uploads, use the largest Telegram photo file bytes.
  - For Telegram video uploads, use the Telegram video thumbnail when available.
  - For URL downloads that are sent back to Telegram as videos, prefer the
    Telegram thumbnail or generated Telegram cover for the sent video.
  - For URL downloads that are sent back to Telegram as photos, use the saved
    photo bytes.
  - For URL failure fallback, prefer Telegram-provided thumbnails. If Telegram
    has no downloadable thumbnail, use the webpage preview image.
- `thumbnail_url`:
  - Store only public webpage preview image URLs.
  - Do not store Telegram file URLs because they can contain bot-token access.

### Lifecycle

- A document is created after the media or fallback thumbnail has been saved and
  the indexable image bytes are available.
- A URL save may fetch page metadata before sending media back to Telegram, but
  the final `image` source can still come from Telegram's returned thumbnail.
- Editing a Telegram media caption updates `caption` only. It must not overwrite
  `title`, `description`, `keywords`, `thumbnail_url`, or URL fields.
- Deleting saved media deletes matching documents by `file_id`.

### Constraints and Indexes

- `belongs_to_id` is required for ownership filtering.
- `inserted_at` remains the default sorting field.
- New URL metadata fields must be optional so existing documents remain valid.
- URL fields are normalized before storage.
- `thumbnail_url` must not contain Telegram Bot API file URLs.

### Migration and Backfill

- Add optional Typesense fields for `title`, `description`, and `keywords`
  before writing them from the bot.
- Existing documents do not need mandatory backfill.
- A best-effort backfill may populate URL metadata for documents that already
  have `url`, but it must tolerate missing pages, private pages, rate limits, and
  provider blocks.

### Retention and Privacy

- Documents are scoped by `belongs_to_id`.
- User-provided `caption` is chat-owned user text and should be shown only in the
  owning chat's search/detail flows.
- Tokenized Telegram file URLs are internal fetch details and must not be stored
  as public metadata.
- Provider metadata may be stale after the source page changes; this index stores
  the metadata observed at save time.

## Relationships

- Telegram message fields provide `caption`, `file_id`, thumbnail files, and
  `source_message_url` inputs.
- URL preview metadata provides `title`, `description`, `keywords`, and public
  preview image URLs.
- Cobalt, `bad.news`, HLS, and direct media downloads provide resolved media
  download URLs and downloaded file bytes.
- Google Drive receives saved media bytes but does not own Typesense metadata.

## Behavior

### Provider Strategy

Metadata extraction is provider-aware. The default provider strategy is reused
unless a platform requires special URL normalization, authenticated fetching, or
field fallback behavior.

Supported provider strategies:

| Platform | URL matching | Default behavior |
| --- | --- | --- |
| `x.com` / old Twitter | `x.com`, `twitter.com`, and subdomains | When `cobalt-cookies.json` contains a `twitter` cookie, fetch authenticated X metadata through the X web GraphQL tweet result endpoint before falling back to public HTML metadata. Store user text in `caption`; store tweet author metadata in `title`; store tweet text or `og:description` in `description`; store hashtags or `meta keywords` in `keywords`; prefer Telegram or generated video thumbnails for `image`. |
| Pinterest | `pinterest.*`, `pin.it` | Use the default URL metadata strategy, with provider-specific normalization allowed for short links. |
| Instagram | `instagram.com` and subdomains | Use the default URL metadata strategy. Provider-specific handling may be added when public metadata is blocked or incomplete. |
| YouTube Shorts | `youtube.com/shorts/*`, `youtu.be/*` when resolved as Shorts | Use the default URL metadata strategy. The Shorts provider may normalize canonical YouTube URLs before metadata fetches. |
| `bad.news` | `bad.news` and subdomains | Use `bad.news` download resolution for media and the default URL metadata strategy for page metadata. |
| `missav.ai` | `missav.ai` and subdomains | Use the default URL metadata strategy first. If the `missav.ai` preview HTML fetch fails, fetch the same path from `https://missav.ws` and store any readable Open Graph metadata while preserving the original `missav.ai` URL as `url`. |
| Other URL | Any supported URL not matched above | Use the default URL metadata strategy. |

### Default URL Metadata Strategy

- Prefer Telegram `link_preview_options.url` as the metadata page URL when
  Telegram provides it.
- Fall back to the original URL.
- Fetch and parse metadata once per save path when possible.
- Extract these fields:
  - `og:title` -> `title`
  - `og:description` -> `description`
  - `meta[name="keywords"]` -> `keywords`
  - `og:image`, `twitter:image`, or video `poster` -> `thumbnail_url` only when
    the URL is public and safe to expose.
- Missing metadata fields are omitted independently; one missing field must not
  block storing the other fields.

## Search

- Text search should include `caption`, `title`, `description`, and `keywords`
  once the fields exist in the Typesense schema.
- `caption` matches should remain high-signal user text matches.
- URL metadata matches should allow finding saves when the user did not write a
  caption.
- Image semantic search continues to use `image_embedding`.
- Search responses and detail views should keep field labels clear so users can
  distinguish their own caption from webpage metadata.
